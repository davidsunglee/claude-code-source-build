# Task System

The task system manages all background work in Claude Code: shell commands running in the background, local and remote sub-agents, in-process teammates, workflow scripts, monitors, and memory-consolidation "dream" agents. Every background unit of work is a **task** -- it has a typed state object stored in `AppState.tasks`, follows a shared lifecycle, and is surfaced to the user through status pills and a task list dialog.

This page covers the type hierarchy, lifecycle, per-variant behavior, UI layer, LLM-facing tools, and the feature gates that control which variants are compiled in.

## Task State Union

All task state is stored in `AppState.tasks` as a `Record<string, TaskState>`. The `TaskState` type is a discriminated union over the `type` field:

```typescript
// source/src/tasks/types.ts
export type TaskState =
  | LocalShellTaskState       // type: 'local_bash'
  | LocalAgentTaskState       // type: 'local_agent'
  | RemoteAgentTaskState      // type: 'remote_agent'
  | InProcessTeammateTaskState // type: 'in_process_teammate'
  | LocalWorkflowTaskState    // type: 'local_workflow'
  | MonitorMcpTaskState       // type: 'monitor_mcp'
  | DreamTaskState            // type: 'dream'
```

Every variant extends `TaskStateBase`, defined in `source/src/Task.ts`:

```typescript
export type TaskStateBase = {
  id: string            // Prefixed ID: 'b' (bash), 'a' (agent), 'r' (remote),
                        //   't' (teammate), 'w' (workflow), 'm' (monitor), 'd' (dream)
  type: TaskType
  status: TaskStatus    // 'pending' | 'running' | 'completed' | 'failed' | 'killed'
  description: string
  toolUseId?: string    // Links back to the tool_use block that spawned this task
  startTime: number
  endTime?: number
  totalPausedMs?: number
  outputFile: string    // Path to on-disk output (symlink or direct file)
  outputOffset: number  // Byte offset for delta reads
  notified: boolean     // Whether completion notification has been sent
}
```

Task IDs are generated by `generateTaskId()`: a single-character type prefix followed by 8 random alphanumeric characters (e.g., `b3k7m2x9q` for a bash task). The 36^8 space (~2.8 trillion) resists brute-force symlink attacks.

The `TaskType` string literals are:

| TaskType | ID Prefix | Variant |
|----------|-----------|---------|
| `local_bash` | `b` | `LocalShellTaskState` |
| `local_agent` | `a` | `LocalAgentTaskState` |
| `remote_agent` | `r` | `RemoteAgentTaskState` |
| `in_process_teammate` | `t` | `InProcessTeammateTaskState` |
| `local_workflow` | `w` | `LocalWorkflowTaskState` |
| `monitor_mcp` | `m` | `MonitorMcpTaskState` |
| `dream` | `d` | `DreamTaskState` |

## Task Lifecycle

Every task follows the same high-level lifecycle:

```
           creation
              |
              v
          +--------+
          | pending |
          +--------+
              |
              v
          +---------+
          | running |
          +---------+
         /     |     \
        v      v      v
 +----------+ +------+ +--------+
 | completed| |failed| | killed |
 +----------+ +------+ +--------+
        \      |      /
         v     v     v
       [notified = true]
              |
              v
         [evicted from
          AppState.tasks]
```

### Status transitions

1. **Creation**: `createTaskStateBase()` sets `status: 'pending'`. Some variants skip straight to `'running'` in their spawn function (e.g., `DreamTask` sets `status: 'running'` at registration).

2. **Pending -> Running**: The spawn function starts the actual work (child process, API call, agent query loop) and transitions to `'running'`.

3. **Running -> Terminal**: One of three terminal states:
   - `'completed'` -- the task finished successfully.
   - `'failed'` -- the task hit an error.
   - `'killed'` -- the task was stopped by the user or system (via `stopTask()` or `Task.kill()`).

4. **Terminal check**: `isTerminalTaskStatus(status)` returns true for `'completed'`, `'failed'`, and `'killed'`. Used to guard against injecting messages into dead tasks.

5. **Notification**: Each task variant sends a `<task-notification>` XML message via `enqueuePendingNotification()` when it reaches a terminal state, then sets `notified: true`.

6. **Eviction**: Once `notified === true` and the status is terminal, `generateTaskAttachments()` (called by `pollTasks()`) adds the task to the eviction list. `evictTerminalTask()` removes the task from `AppState.tasks`, freeing memory. `LocalAgentTask` has a grace period (`PANEL_GRACE_MS = 30s`) and a `retain` flag that blocks eviction while the UI is viewing the task.

### The `Task` interface

Each variant exports a `Task` object implementing:

```typescript
export type Task = {
  name: string
  type: TaskType
  kill(taskId: string, setAppState: SetAppState): Promise<void>
}
```

The `kill` method is the only polymorphic dispatch point. `spawn` and `render` are not called polymorphically -- they are type-specific entry points.

### Task registry

`getAllTasks()` in `source/src/tasks.ts` returns the live array of `Task` implementations. `LocalWorkflowTask` and `MonitorMcpTask` are conditionally included via `bun:bundle` feature gates (`WORKFLOW_SCRIPTS` and `MONITOR_TOOL`). `getTaskByType()` looks up a task by its `TaskType` string, used by `stopTask()` to dispatch `kill()`.

## Per-Variant Details

### LocalShellTask

**Type**: `local_bash` | **ID prefix**: `b` | **Feature gate**: always on

Runs a shell command in the background. Created when the LLM uses `BashTool` with `run_in_background: true`.

**Key state fields** (beyond `TaskStateBase`):
- `command: string` -- the shell command being run.
- `result?: { code: number; interrupted: boolean }` -- exit result.
- `shellCommand: ShellCommand | null` -- handle to the running child process.
- `isBackgrounded: boolean` -- `false` while still in the foreground spawning phase; `true` once backgrounded.
- `agentId?: AgentId` -- the agent that spawned this task (undefined = main thread).
- `kind?: 'bash' | 'monitor'` -- UI display variant. `'monitor'` shows description instead of command.

**Kill behavior**: Sends SIGTERM to the child process, calls `shellCommand.cleanup()`, clears timers, sets status to `'killed'`.

**Orphan cleanup**: `killShellTasksForAgent()` kills all running shell tasks spawned by a given agent when that agent exits, preventing zombie processes from outliving their parent.

**Stall detection**: A watchdog timer checks every 5 seconds if output has stopped growing. After 45 seconds of silence, if the last output line matches interactive-prompt patterns (e.g., `(y/n)`, `Press Enter`), it sends a notification advising the model to kill and retry with piped input.

**Files**: `LocalShellTask/LocalShellTask.tsx`, `LocalShellTask/guards.ts`, `LocalShellTask/killShellTasks.ts`

---

### LocalAgentTask

**Type**: `local_agent` | **ID prefix**: `a` | **Feature gate**: always on

Runs a sub-agent (via `AgentTool`) in a separate query loop within the same process. This is the primary mechanism for parallel work in Claude Code.

**Key state fields**:
- `agentId: string` -- unique agent identifier.
- `prompt: string` -- the initial prompt sent to the agent.
- `selectedAgent?: AgentDefinition` -- the agent definition, if using a named agent.
- `agentType: string` -- e.g., `'worker'`, `'main-session'`.
- `model?: string` -- optional model override.
- `abortController?: AbortController` -- for cancellation.
- `progress?: AgentProgress` -- live progress: `toolUseCount`, `tokenCount`, `lastActivity`, `recentActivities`.
- `result?: AgentToolResult` -- the agent's final output.
- `messages?: Message[]` -- conversation history for the zoomed transcript view.
- `isBackgrounded: boolean` -- whether the task is in the background pill vs. foreground.
- `pendingMessages: string[]` -- messages queued via `SendMessage`, drained at tool-round boundaries.
- `retain: boolean` -- blocks eviction while UI is viewing the task.
- `diskLoaded: boolean` -- whether the sidechain JSONL has been bootstrapped into `messages`.
- `evictAfter?: number` -- timestamp deadline for panel visibility.

**Progress tracking**: `ProgressTracker` accumulates `toolUseCount`, `latestInputTokens` (cumulative per-turn), and `cumulativeOutputTokens`. `updateProgressFromMessage()` updates it from each assistant message. `getProgressUpdate()` converts it to the `AgentProgress` shape stored on the task.

**Kill behavior**: Aborts the `AbortController`, cleans up, sets status to `'killed'`.

**Also used for**: The backgrounded main session (`LocalMainSessionTask`) reuses `LocalAgentTaskState` with `agentType: 'main-session'` and an `'s'` ID prefix.

**Files**: `LocalAgentTask/LocalAgentTask.tsx`, `LocalMainSessionTask.ts`

---

### InProcessTeammateTask

**Type**: `in_process_teammate` | **ID prefix**: `t` | **Feature gate**: always on (requires agent swarms to be enabled at runtime)

Runs an in-process teammate -- a sub-agent that shares the Node.js process using `AsyncLocalStorage` for isolation. Part of the "swarm" / team system.

**Key state fields**:
- `identity: TeammateIdentity` -- contains `agentId` (e.g., `"researcher@my-team"`), `agentName`, `teamName`, `color`, `planModeRequired`, `parentSessionId`.
- `prompt: string` -- initial task prompt.
- `awaitingPlanApproval: boolean` -- whether the teammate is waiting for plan approval.
- `permissionMode: PermissionMode` -- cycled independently via Shift+Tab when viewing.
- `messages?: Message[]` -- capped at `TEAMMATE_MESSAGES_UI_CAP = 50` entries to control memory. The full conversation lives in the local `allMessages` array in `inProcessRunner` and on disk.
- `pendingUserMessages: string[]` -- queue of user messages to deliver.
- `isIdle: boolean` -- whether the teammate is waiting for work.
- `shutdownRequested: boolean` -- signals the teammate to exit.
- `onIdleCallbacks?: Array<() => void>` -- runtime callbacks for efficient wait-without-polling.

**Memory management**: BQ analysis found ~20MB RSS per agent at 500+ turns and ~125MB per concurrent agent in swarm bursts. The `TEAMMATE_MESSAGES_UI_CAP` limit and `appendCappedMessage()` helper prevent the AppState mirror from growing unbounded.

**Kill behavior**: Delegates to `killInProcessTeammate()` in `utils/swarm/spawnInProcess.ts`.

**Files**: `InProcessTeammateTask/InProcessTeammateTask.tsx`, `InProcessTeammateTask/types.ts`

---

### RemoteAgentTask

**Type**: `remote_agent` | **ID prefix**: `r` | **Feature gate**: always on

Runs a task on a remote Claude.ai session. Used for cloud-offloaded work including ultraplan and ultrareview.

**Key state fields**:
- `sessionId: string` -- the remote session ID for API calls.
- `remoteTaskType: RemoteTaskType` -- one of `'remote-agent'`, `'ultraplan'`, `'ultrareview'`, `'autofix-pr'`, `'background-pr'`.
- `remoteTaskMetadata?: RemoteTaskMetadata` -- task-specific metadata (e.g., PR number, repo).
- `command: string`, `title: string` -- human-readable descriptions.
- `todoList: TodoList` -- structured task tracking.
- `log: SDKMessage[]` -- accumulated session messages.
- `isLongRunning?: boolean` -- suppresses completion after first `result`.
- `pollStartedAt: number` -- when the local poller started, used for timeout calculation so resumes don't immediately expire.
- `isRemoteReview?: boolean` -- created by `/ultrareview`.
- `reviewProgress?: { stage, bugsFound, bugsVerified, bugsRefuted }` -- parsed from heartbeat echoes.
- `isUltraplan?: boolean` -- ultraplan mode flag.
- `ultraplanPhase?: 'needs_input' | 'plan_ready'` -- scanner-derived pill state.

**Completion checkers**: `registerCompletionChecker()` allows registering a callback for a `RemoteTaskType`. Invoked on every poll tick; returns a string to complete the task, or `null` to keep polling.

**Metadata persistence**: Remote agent metadata is written to a session sidecar file so that `--resume` can restore polling for tasks that were running when the session ended.

**Kill behavior**: Aborts the session, archives it, removes sidecar metadata, emits SDK terminated event.

**Files**: `RemoteAgentTask/RemoteAgentTask.tsx`

---

### DreamTask

**Type**: `dream` | **ID prefix**: `d` | **Feature gate**: `KAIROS_DREAM` (or `KAIROS`)

Runs the auto-dream memory consolidation agent. Makes the forked dream agent visible in the footer pill and task dialog. The dream agent itself is unchanged; this is pure UI surfacing.

**Key state fields**:
- `phase: 'starting' | 'updating'` -- flips to `'updating'` when the first Edit/Write tool_use is observed.
- `sessionsReviewing: number` -- how many sessions the dream agent is reviewing.
- `filesTouched: string[]` -- paths observed in Edit/Write tool_use blocks (incomplete; misses bash-mediated writes).
- `turns: DreamTurn[]` -- assistant text responses with tool use counts, capped at 30.
- `priorMtime: number` -- stashed so `kill` can rewind the consolidation lock.

**Kill behavior**: Aborts the `AbortController`, then calls `rollbackConsolidationLock(priorMtime)` so the next session can retry the dream.

**Notification**: `notified: true` is set immediately on completion/failure/kill. Dream has no model-facing notification path -- it is UI-only.

**Files**: `DreamTask/DreamTask.ts`

---

### LocalWorkflowTask

**Type**: `local_workflow` | **ID prefix**: `w` | **Feature gate**: `WORKFLOW_SCRIPTS`

Runs a local workflow script. The implementation is behind the `WORKFLOW_SCRIPTS` feature gate and is conditionally `require()`-d in `source/src/tasks.ts`. Not present in the open source build.

**Pill label**: `"1 background workflow"` / `"N background workflows"`.

---

### MonitorMcpTask

**Type**: `monitor_mcp` | **ID prefix**: `m` | **Feature gate**: `MONITOR_TOOL`

Runs a monitor via MCP. The implementation is behind the `MONITOR_TOOL` feature gate and is conditionally `require()`-d in `source/src/tasks.ts`. Not present in the open source build.

When `MONITOR_TOOL` is enabled, it also changes behavior in `LocalShellTask`: monitor-kind shell tasks get `'next'` priority for notifications (drained without a Sleep barrier) and have their stall watchdog disabled.

**Orphan cleanup**: When an agent exits, `MonitorMcpTask.killMonitorsForAgent()` is called (gated behind `feature('MONITOR_TOOL')` in `runAgent.ts`) to clean up monitors scoped to that agent.

**Pill label**: `"1 monitor"` / `"N monitors"`.

## Task UI

### Status pills

The footer pill displays a compact summary of active background tasks. `getPillLabel()` in `source/src/tasks/pillLabel.ts` computes the label text based on the task types:

| Task type(s) | Pill label examples |
|---|---|
| `local_bash` | `"1 shell"`, `"2 shells"`, `"1 monitor"`, `"1 shell, 2 monitors"` |
| `local_agent` | `"1 local agent"`, `"3 local agents"` |
| `remote_agent` | `"◇ 1 cloud session"`, `"◇ 2 cloud sessions"` |
| `remote_agent` (ultraplan) | `"◇ ultraplan"`, `"◆ ultraplan ready"`, `"◇ ultraplan needs your input"` |
| `in_process_teammate` | `"1 team"`, `"2 teams"` (counted by distinct `teamName`) |
| `local_workflow` | `"1 background workflow"` |
| `monitor_mcp` | `"1 monitor"` |
| `dream` | `"dreaming"` |
| Mixed types | `"N background tasks"` |

`pillNeedsCta()` returns true for single ultraplan tasks with an active `ultraplanPhase`, causing the pill to show a dimmed `" . down-arrow to view"` call-to-action.

### Background task filtering

`isBackgroundTask()` determines whether a task appears in the background pill. A task qualifies if:
1. Its status is `'running'` or `'pending'`.
2. It has been explicitly backgrounded (`isBackgrounded` is not `false`).

Tasks that are still in the foreground spawning phase (`isBackgrounded === false`) are excluded.

### Task list component

`TaskListV2` (`source/src/components/TaskListV2.tsx`) renders the task list dialog (accessed via Shift+Down). It shows tasks sorted by ID, limits display to `min(10, rows - 14)` entries, and tracks recently-completed tasks for a 30-second fade-out period.

### Foregrounding

Tasks can be foregrounded from the task list. For main-session tasks, `foregroundMainSessionTask()` swaps `isBackgrounded` to `false` and sets `foregroundedTaskId` on `AppState`. For teammates, the UI enters a zoomed transcript view.

## Task Tools

These are LLM-facing tools for managing a structured task list (the "TodoV2" system). They are distinct from the background task infrastructure described above -- these tools manage a file-backed checklist of work items, not background processes.

| Tool name | Constant | Purpose |
|-----------|----------|---------|
| `TaskCreate` | `TASK_CREATE_TOOL_NAME` | Create a new task with subject, description, optional activeForm and metadata |
| `TaskUpdate` | `TASK_UPDATE_TOOL_NAME` | Update task subject, description, status, blocking relationships, owner, metadata |
| `TaskGet` | `TASK_GET_TOOL_NAME` | Retrieve a single task by ID |
| `TaskList` | `TASK_LIST_TOOL_NAME` | List all tasks with their statuses |
| `TaskStop` | `TASK_STOP_TOOL_NAME` | Stop a running background task by ID (this one bridges both systems) |
| `TaskOutput` | `TASK_OUTPUT_TOOL_NAME` | Get output from a background task, optionally blocking until completion with configurable timeout |

### TaskStop

`TaskStop` (aliased as `KillShell` for backward compatibility) is the bridge between the LLM and the background task system. It delegates to `stopTask()` in `source/src/tasks/stopTask.ts`, which:

1. Looks up the task in `AppState.tasks`.
2. Validates the task is running.
3. Finds the `Task` implementation via `getTaskByType()`.
4. Calls `task.kill()`.
5. For shell tasks, suppresses the "exit code 137" notification (noise) and emits a direct SDK event instead.

### TaskOutput

`TaskOutput` lets the LLM read output from any background task type. It supports blocking (`block: true`, default) with a configurable timeout (default 30s, max 600s). Handles all task types uniformly through a `getTaskOutputData()` function.

### TodoV2 tools (TaskCreate, TaskUpdate, TaskGet, TaskList)

These are enabled when `isTodoV2Enabled()` returns true. They manage a file-backed task list (separate from `AppState.tasks`) with statuses like `open`, `in_progress`, `completed`, etc. In-process teammates get additional access to these tools via `IN_PROCESS_TEAMMATE_ALLOWED_TOOLS`.

## Coordinator Mode

Coordinator mode transforms the main Claude Code session into an orchestrator that delegates all substantive work to sub-agent workers. It is gated behind the `COORDINATOR_MODE` feature flag and activated at runtime by the `CLAUDE_CODE_COORDINATOR_MODE` environment variable.

### Activation

```typescript
// source/src/coordinator/coordinatorMode.ts
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
  }
  return false
}
```

### Tool filtering

When coordinator mode is active, the coordinator's tool set is restricted to:

```typescript
// source/src/constants/tools.ts
export const COORDINATOR_MODE_ALLOWED_TOOLS = new Set([
  AGENT_TOOL_NAME,        // 'Task' (AgentTool)
  TASK_STOP_TOOL_NAME,    // 'TaskStop'
  SEND_MESSAGE_TOOL_NAME, // 'SendMessage'
  SYNTHETIC_OUTPUT_TOOL_NAME,
])
```

The coordinator cannot use Bash, Read, Edit, or other direct tools. Workers spawned by the coordinator get the full `ASYNC_AGENT_ALLOWED_TOOLS` set.

### System prompt

`getCoordinatorSystemPrompt()` returns a detailed system prompt that instructs the model to:
- Orchestrate work across parallel workers.
- Synthesize research findings before delegating implementation.
- Use `AgentTool` to spawn workers, `SendMessage` to continue them, and `TaskStop` to stop them.
- Follow a phased workflow: Research (parallel) -> Synthesis (coordinator) -> Implementation -> Verification.

### Session restore

`matchSessionMode()` detects mismatches between the current mode and a resumed session's stored mode. It flips the `CLAUDE_CODE_COORDINATOR_MODE` env var to match the session, ensuring resumed sessions behave consistently regardless of how the CLI was launched.

### User context

`getCoordinatorUserContext()` injects context about worker capabilities: which tools workers can use, which MCP servers are connected, and the scratchpad directory path (if enabled).

## Feature Gates

The task system uses four `bun:bundle` compile-time feature gates. When a gate is off, the corresponding code is eliminated at build time via dead-code elimination (conditional `require()` calls that resolve to `null`).

| Gate | Controls | Effect when off |
|------|----------|-----------------|
| `COORDINATOR_MODE` | Coordinator mode: tool filtering, system prompt, session mode persistence, worker context injection | `isCoordinatorMode()` always returns `false`; coordinator-specific imports are eliminated |
| `WORKFLOW_SCRIPTS` | `LocalWorkflowTask`, the `/workflows` command, `WorkflowTool`, workflow tool disallowed-list entry | `LocalWorkflowTask` is `null` in `getAllTasks()`; workflow commands are not registered |
| `MONITOR_TOOL` | `MonitorMcpTask`, monitor-kind shell task behavior, `sleep N` blocking in BashTool, agent-scoped monitor cleanup | `MonitorMcpTask` is `null` in `getAllTasks()`; bash sleep blocking is not enforced |
| `KAIROS_DREAM` (or `KAIROS`) | `DreamTask` registration in the skill system, dream skill registration | Dream skill is not registered; `DreamTask` itself is always compiled in but only spawned when the dream skill fires |

Note that `DreamTask` is always included in `getAllTasks()` regardless of feature gates. The `KAIROS_DREAM` gate controls the *skill* that triggers dream consolidation, not the task implementation itself.
